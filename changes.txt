1. CRITICAL BUG FIX: benchmark_data missing from AgentStatePatchModel
--------------------------------------------------------------------------------
File: agent/state_models.py

Problem:
  The benchmark_node was added to the graph but `benchmark_data` was never
  declared in AgentStatePatchModel or AgentStateModel (both use extra="forbid").
  Once the benchmark node populated state with `benchmark_data`, EVERY
  downstream node that validates input against AgentStatePatchModel failed
  contract validation — including synthesis_gate. The observability wrapper
  degraded synthesis_gate, returning {} instead of running it, so
  `final_response` was never set. The frontend showed:
    "Pipeline did not produce final_response."

Fix:
  Added `benchmark_data: NodeEnvelope[dict[str, Any]] | None = None` to both
  AgentStateModel and AgentStatePatchModel.

Impact:
  This was the root cause of synthesis always appearing "blocked" even when
  data was valid. All downstream nodes (risk, prioritization, pipeline_status,
  signal_enrichment, synthesis_gate) were silently degrading.

2. HARD GATING MECHANISM (Gates 5-8)
--------------------------------------------------------------------------------
File: agent/nodes/synthesis_gate.py

Added `pre_synthesis_audit(state)` — a unit-testable function that enforces
minimum data requirements BEFORE synthesis is allowed.

Gate 5: Data Depth
  - If KPI time-series depth < 3 data points -> block output
  - Prevents insights from statistically meaningless samples

Gate 6: Analytical Layer Count (softened by Section 7)
  - If valid scoring layers == 0 -> hard block (no usable signal at all)
  - If valid scoring layers 1-2 -> partial_insight (synthesis allowed, degraded)
  - If valid scoring layers >= 3 -> full_insight

Gate 7: KPI Coverage
  - If KPI metric coverage < 50% of expected metrics -> block output
  - Prevents insights built on incomplete metric sets

Gate 8: Forecast + Alternatives (softened by Section 7)
  - If forecast is unusable AND zero valid layers -> block
  - If forecast is unusable BUT 1+ valid layers exist -> partial_insight (not block)

Return structure when blocked:
  {
    "status": "blocked",
    "reason": "depth=2; valid_layers=1",
    "missing_requirements": [
      "insufficient data depth (2 < 3 required)",
      "insufficient analytical layers (1 < 3 required)"
    ],
    "uncertainty_mode": false,
    "conflict_summary": null,
    "audit_details": {
      "kpi_depth": 2,
      "valid_layer_count": 1,
      "kpi_coverage_ratio": 0.25,
      "forecast_usable": false,
      "overall_confidence": 0.15
    }
  }

Integration:
  - `should_block_synthesis()` calls `pre_synthesis_audit()` as Gates 5-8
    (after existing Gates 1-4)
  - `build_blocked_response()` includes `missing_requirements` in the
    failure reason text
  - `synthesis_gate_node()` includes `missing_requirements`,
    `uncertainty_mode`, and `conflict_summary` in envelope_diagnostics

Constants added:
  _MIN_DEPTH = 3
  _MIN_VALID_LAYERS = 3
  _MIN_KPI_COVERAGE = 0.5

4. ISOLATION ENFORCEMENT (Forecast Signal Filtering)
--------------------------------------------------------------------------------
Files: agent/nodes/signal_conflict_node.py
       agent/nodes/risk_node.py

Problem:
  Forecast layer correctly marked as gate_status="isolate" (due to low R²)
  but its signals (slope, deviation, etc.) still influenced:
    - conflict detection (producing spurious conflicts)
    - risk scoring (inflating/deflating risk based on unreliable forecast)
    - prioritization (via cascading conflict/risk data)

Solution:
  Added strict isolation enforcement at two downstream consumption points:

  a) signal_conflict_node.py:
     - At the start of _collect_global_signals(), computes layer integrity

     - Adds warning: "forecast signals excluded from conflict detection:
       layer isolated"
     - Logs isolation enforcement for debugging
     - Records isolated_layers in metadata

  b) risk_node.py:
     - Moved integrity computation to early in the node (before forecast
       signal extraction) instead of late (after risk scoring)
     - Checks if "forecast" is in isolated_layers
     - If isolated: treats forecast as unavailable — falls through to
       growth/cohort/scenario proxy path
     - Logs: "Forecast layer isolated - excluding forecast signals from
       risk scoring"
     - Reuses early integrity computation later (no duplicate compute)

  Import added:
     signal_conflict_node.py now imports UnifiedSignalIntegrity

5. CONFIDENCE SCORING REDESIGN
--------------------------------------------------------------------------------
File: agent/signal_integrity.py

Problem:
  Final confidence scores were inflated (e.g., ~0.64) even when:
    - model confidence = 0
    - forecast unusable
    - missing signals
    - degraded layers
  The old system had no caps — a single healthy layer could produce
  misleadingly high confidence.

Solution:
  Added monotonic confidence caps after the authority-weighted scoring.
  More missing/degraded signals -> strictly lower confidence (never inflated).

  Caps applied sequentially (most restrictive wins):

  Cap 1: KPI layer degraded -> confidence <= 0.35
    - Critical layer weakness must be reflected in score

  Cap 2: Forecast unusable (isolated or blocked) -> confidence <= 0.30
    - Cannot claim high confidence without predictive signal

  Cap 3: Insufficient analytical layers (< 3 valid) -> confidence <= 0.25
    - Single-dimensional analysis cannot support strong conclusions

  Transparent breakdown added to compute() return:
  {
    "confidence_breakdown": {
      "base": 0.64,          // score before caps
      "penalties": {
        "kpi_penalty_applied": false,
        "caps_applied": [
          {"cap": "insufficient_layers", "limit": 0.25, "valid_layers": 1}
        ]
      },
      "final": 0.25           // score after all caps
    }
  }

  Convenience fields added to compute() return:
    "kpi_depth": int           // time-series data points
    "kpi_coverage_ratio": float // fraction of expected metrics present
    "valid_layer_count": int    // number of scoring layers
    "forecast_usable": bool     // forecast layer pass or degrade (not isolated/blocked)

  Caps generate reasoning_warnings for auditability:
    "Only 1 valid layer(s): confidence capped at 0.25"
    "Forecast unusable (isolate): confidence capped at 0.3"
    "KPI layer degraded: confidence capped at 0.35"

  Guarantees:
    - Monotonic: more missing -> lower score (caps only reduce, never increase)
    - base >= final (invariant)
    - No arbitrary boosts
    - Breakdown always present for transparency

7. MULTI-LAYER REASONING INTEGRITY (Insight Quality Classification)
--------------------------------------------------------------------------------
Files: agent/signal_integrity.py
       agent/nodes/synthesis_gate.py
       agent/nodes/prioritization_node.py

Problem:
  The system had only two output modes: "allowed" or "blocked".  States with
  1-2 valid analytical layers were either silently passed through (producing
  overconfident insights) or hard-blocked (producing no output at all).

Solution:
  Added three-state insight quality classification:

  a) signal_integrity.py — insight_quality field in compute():
     - full_insight  (3+ valid layers): full recommendations allowed
     - partial_insight (1-2 valid layers): synthesis allowed but degraded
     - blocked (0 valid layers or KPI hard-failed): synthesis blocked

     Added layer_classification dict:
       Each of 5 layers (kpi, forecast, competitive, cohort, segmentation)
       classified as: valid | degraded | isolated | missing | inactive

  b) synthesis_gate.py — Gate 6 softened:
     - Old: valid_layer_count < 3 → hard block
     - New: valid_layer_count == 0 → hard block; 1-2 → partial_insight (degraded)

     Gate 8 softened:
     - Old: forecast_unusable AND valid_layers < 3 → block
     - New: forecast_unusable AND valid_layers == 0 → block
     - 1-2 valid layers with unusable forecast → partial_insight

     Gate 3 made insight_quality-aware:
     - partial_insight states bypass the confidence threshold check (0.4)
       since their confidence is already capped at 0.25 by design

  c) prioritization_node.py — partial_insight handling:
     - If insight_quality == "partial_insight":
       → recommended_focus replaced with:
         "partial insight — insufficient analytical layers for
          recommendations; additional data sources required"
       → priority_level forced to "low"
       → decision set to "withheld"
     - Added insight_quality and layer_classification to output dict

8. MISSING SIGNAL DETECTION
--------------------------------------------------------------------------------
File: agent/signal_integrity.py

Problem:
  Missing required signals were silently defaulting to zero values.  No
  structured report existed to tell downstream consumers which signals were
  absent, whether they were required or optional, and what impact their
  absence had on confidence.

Solution:
  Added _detect_missing_signals() classmethod to UnifiedSignalIntegrity:

  a) Structured missing signal report:
     {
       "missing_signals": [
         {
           "signal": "risk",
           "state_key": "risk_data",
           "classification": "required",
           "status": "missing",
           "impact": "confidence penalty"
         },
         {
           "signal": "forecast",
           "state_key": "forecast_data",
           "classification": "optional",
           "status": "missing",
           "impact": "reduced coverage"
         }
       ],
       "required_missing_count": 1,
       "optional_missing_count": 1,
       "actions_taken": [
         "required signal 'risk' missing: confidence penalised",
         "optional signal 'forecast' missing: coverage reduced"
       ]
     }

  b) Signal classification:
     Uses graph_config.py's GraphNodeConfig to classify each signal as
     "required" or "optional" based on business type.

  c) Confidence penalty for missing required signals:
     Multiplicative penalty: factor = max(0.50, 1.0 - 0.15 * count)
     - 1 missing → 0.85x
     - 2 missing → 0.70x
     - 3+ missing → 0.50x (floor)
     Multiplicative avoids zeroing already-low scores from other caps.
     Penalty recorded in caps_applied and reasoning_warnings.

  d) Detection coverage:
     - Checks all required + optional state keys from GraphNodeConfig
     - Detects missing (None), failed, and skipped signals
     - Cross-checks against layer-level availability from integrity model
     - No silent defaults — every missing signal explicitly reported

  e) Integration:
     - missing_signal_report added to compute() return dict
     - Surfaced in synthesis_gate_node() envelope_diagnostics
     - Penalty applied after confidence caps (stacks correctly)

     - competitive_context → llm_node → prompt (numeric_signals)
     - Both reach the LLM; neither silently drops

17. FIX: LLM VALIDATION FAILURE ON CONSTRAINED BENCHMARK DATA
--------------------------------------------------------------------------------
Files: agent/nodes/llm_node.py
       llm_synthesis/schema.py
       llm_synthesis/prompt_builder.py

Problem:
  When competitive data was constrained (partial benchmark, thin peers), the
  LLM generated generic recommendations that failed the StrategicRecommendations
  specificity validator ("Each recommendation must reference specific context").
  After 3 retries, the system fell back to failure() which produced:
    "competitor benchmark analysis could not be completed due to competitor
     data constraints"
  This happened because:
  a) The bridge from benchmark_data to competitor mode activated on thin data
     (2+ peers was enough), switching to competitor prompt even when the LLM
     didn't have enough specifics to write valid recommendations.
  b) _ACCEPTED_RECOMMENDATION_TERMS didn't include benchmark-specific terms
     (benchmark, peer, percentile, position) so benchmark-referenced
     recommendations failed validation.
  c) _public_failure_reason stripped the actual validation error details,
     so failure() couldn't route to the correct template.

19. FIX: SYNTHESIS BLOCKED AT 0.05 — TWO ROOT CAUSES

Fix A — Auto-enable when user provides competitors:
  competitor_node now checks state["competitors"] (user-provided list).
  If the list is non-empty, the node enables itself automatically,
  bypassing the COMP_INTEL_ANALYZE_ENABLED requirement.

Root cause B — Hard gates too aggressive for sparse data:
  Gates 5 and 7 in pre_synthesis_audit treated ALL threshold violations
  as hard blocks (status="blocked"), even when the user had SOME data:
  - depth=2 (< 3 ideal) → blocked (zero output)
  - coverage=25% (< 50% ideal) → blocked (zero output)

  This created a dead zone: users with limited but real data could never
  get any output, even with degraded/hedged tone.

Fix B — Differentiate hard blocks from soft blocks:
  - depth == 0 or coverage == 0% → hard block (genuinely no data)
  - depth 1-2 or coverage > 0% but < 50% → soft block (degraded synthesis)
  - Soft blocks return status="degraded" with insight_quality="partial_insight"
  - partial_insight bypasses the confidence threshold check (Gate 3)
  - Synthesis proceeds with hedged tone, not blocked entirely
